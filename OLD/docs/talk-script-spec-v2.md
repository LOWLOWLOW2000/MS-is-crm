# トークスクリプト仕様 v2（変更案）テキストまとめ・整合性チェック

## 1. 新仕様のテキストまとめ

### 1.1 表示形式（ABC）

- **トークスクリプトは「ABC」という形で表示する。**
  - メインタブを A / B / C の区分（または階層）で表示する想定。
  - ※「ABC」の具体的な意味（例: A=オープニング、B=本論、C=締め など）は要件確認で確定する。

### 1.2 メインタブとサブタブの関係

- **各スクリプト（メインタブ）を選択すると、そのタブの直下にサブタブが出現する。**
  - 1段目: メインタブ（ABC など）
  - 2段目: 選択したメインタブに紐づくサブタブ

### 1.3 サブタブの構成（架電フロー用）

サブタブは次の4種を想定（表記は要望文のまま。必要に応じて「本人フロント」は本人対応/フロント、「out返し」は反論返しと読み替え可）。

| サブタブ       | 内容・特記事項 |
|----------------|----------------|
| 受付突破       | 既存の受付突破トーク |
| 本人フロント   | （導入・フロント相当） |
| 反論返し       | 反論対応・切り返し |
| クロージング   | アポ条件チェックボックスあり。**アポと紐付けて記録し、条件回収率として集計する** |

- **クロージング**
  - 画面上に「アポ条件」のチェックボックスを設ける。
  - チェック結果は**アポ（商談・アポ獲得）と紐付けて記録**する。
  - 集計指標として**条件回収率**を算出・利用する。

### 1.4 商品説明タブの特別扱い

- **商品説明**では以下を行う。
  - **クライアント情報**もあわせて表示する。
  - **PJインフォ**: 商品説明タブ選択時、その**下にサブタブが出現**する。
    - 例: 「PJのクライアント情報」「商品情報」など、ディレクターが登録した項目をタブで選択可能。

### 1.5 PJインフォの登録・管理（ディレクター設定）

- **登録**
  - ディレクター（または管理者）が「PJインフォ」を登録する画面・機能を用意する。
  - 登録形式: **PDF添付** および **テキスト**。
  - ※要望の「PDF圧縮」は **PDF添付** と解釈（圧縮は実装時に要否を検討）。

- **サブタブの設定**
  - PJインフォの各項目（サブタブ）は**任意の名称**を設定できる。
  - 一覧から**登録**と**非表示**の操作ができる。
  - **削除機能は持たない**。非表示にした項目は「非表示一覧」に格納し、必要に応じて再表示できる。

## 2. 現行仕様との対応・差分

| 項目           | 現行（.cursorrules / 実装） | 新仕様 v2 |
|----------------|----------------------------|-----------|
| タブ階層       | 1段のみ（企業HP + 固定タブ + 自由書式） | 2段（メインABC + サブタブ） |
| 固定タブ       | 受付突破・導入・反論・ヒアリング・クロージング・商品説明 | メインをABC表示、サブに 受付突破・本人フロント・反論返し・クロージング |
| 商品説明       | 本文のみ                    | クライアント情報表示 + PJインフォ（サブタブ、PDF/テキスト登録） |
| クロージング   | 本文のみ                    | アポ条件チェックボックス、アポ紐付け、条件回収率 |

## 3. 整合性チェック

### 3.1 用語・表記

- **本人フロント**: 現行の「導入トーク」に相当する可能性が高い。実装時に「本人フロント」のままにするか「導入」に統一するか決定する。
- **out返し**: 「反論返し」と解釈。反論対応と同じ扱いで矛盾なし。
- **PDF圧縮**: 登録方法として「PDF添付」「テキスト」と解釈。圧縮はストレージ対策として別検討でよい。

### 3.2 データ

- **アポ条件・条件回収率**
  - クロージングのチェックボックス値を「アポ」に紐付けて記録するため、**アポ（商談/約束）を識別するエンティティまたはレコード**が存在する前提となる。
  - 現行の架電記録（結果・メモ・次回架電）に「アポ」フラグやアポIDを拡張する、もしくは別テーブルでアポを管理する必要がある。いずれかで一貫していれば整合可。

### 3.3 権限

- PJインフォの登録・非表示を「ディレクターが設定できる」とあるため、**ディレクター（および is_admin 等）がスクリプト/PJインフォを編集可能**とする現行の権限設計と整合する。

### 3.4 削除と非表示

- 「削除はなく非表示を設定すると非表示一覧に格納」は、論理削除（isVisible / deletedAt 等）で実装すれば整合。物理削除しない方針で一貫。

### 3.5 結論

- 上記の解釈（ABC=メイン区分、サブタブ=受付突破・本人フロント・反論返し・クロージング、商品説明=PJインフォ+クライアント情報、クロージング=アポ条件・条件回収率、PJインフォ=PDF/テキスト・任意名称・非表示一覧）で**矛盾はなし**。
- 実装時には以下を確定する必要あり:
  - ABC の具体的なラベル・内容（A/B/C の定義）
  - メインタブとサブタブの対応（どのメインにどのサブを表示するか）
  - アポをどのテーブル/APIで管理するか
  - 自由書式・タブ上限の扱い（現行を残すか、新仕様に合わせて変更するか）
