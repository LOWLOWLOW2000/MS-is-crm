# タスク: 架電キャッシュ方針（バックエンド議論用）

## 背景

架電画面ではリスト内容確認・修正セーブ・承認フローでキャッシュ（Zustand + localStorage）を利用している。  
「キャッシュがめちゃくちゃ溜まる」ため、以下を検討するタスクとして話し合う。

## 検討したい項目

1. **定期的にクリアする頻度**
   - 例: 日次 / セッション終了時 / リスト切り替え時 / 一定件数保存ごと
   - フロントのみで完結するか、バックエンドの TTL やキューと連動するか

2. **保持するキャッシュサイズの上限**
   - 件数上限（例: 直近 N 件の下書きのみ保持）
   - ストレージ容量ベースの上限（localStorage の残量考慮）
   - テナント／ユーザーごとの上限をバックエンドで持つか

3. **バックエンドとの役割分担**
   - 下書き・承認状態をサーバーに持つか、クライアントのみか
   - キャッシュ無効化・一括クリアの API 要否
   - 条件回収率・アポ条件など、保存データとキャッシュの整合性

## 関連

- 架電専用UI: `apps/web/app/(dashboard)/calling/page.tsx`
- セッションストア: `apps/web/lib/stores/calling-session-store.ts`（persist: localStorage）
- .cursorrules の「架電キャッシュ設計」
