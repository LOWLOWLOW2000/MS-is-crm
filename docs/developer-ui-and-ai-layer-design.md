# 開発者UIと営業AIレイヤーの設計メモ（話し合い用）

※OLチェック・モック閲覧とは無関係。今後の構想・設計の議論用ドキュメント。

---

## 0. 言葉定義（プロジェクト名）

| 呼び方 | プロジェクト名 | 通称 | 説明 |
|--------|----------------|------|------|
| SaaS 側 | **MG_SFA01** | **SFA** | 架電管理・リスト・レポート・権限・テナント・課金など「プロダクト」として提供する部分。いまの is-crm がここに相当。 |
| 営業AI 側 | **MG_AI-kinai** | **アキナイ** | 他社AI（LLM・音声・評価など）を組み合わせて作る「営業用AI」の部分。別プロジェクトとして開発し、**最終的にデプロイする単位も SFA と分ける**。 |

- **「アプリが2個」のイメージ**  
  → **あっている**。最終的に **配布・デプロイする「アプリ」は2つ**：  
  1. **SFA（MG_SFA01）** … ユーザーが触るCRM（Web＋API＋DB など）  
  2. **アキナイ（MG_AI-kinai）** … 営業AI 用のサービス（API・バッチ・開発者UI など）  
  SFA は「アキナイを呼ぶ窓口」だけ持ち、中身（モデル・プロンプト・パラメータ）はアキナイ側に閉じる。

---

## 1. 前提：SFA と アキナイ は切り離す

- **SFA（MG_SFA01）**  
  架電管理・リスト・レポート・権限・テナント・課金など「 product 」として提供する部分。
- **アキナイ（MG_AI-kinai）**  
  他社のAI（LLM・音声・評価など）を組み合わせて「営業用AI」として設計する部分。**別プロジェクトとして最終でデプロイ**する。  
  → **SFA と アキナイ は切り離したほうがよい**、という方針で話を進める想定。

**切り離しのイメージ**

- **SFA**：UI・API・DB・認証・マルチテナント。AI は「ブラックボックスなサービス（アキナイ）」として呼ぶだけ。
- **アキナイ**：プロンプト設計・モデル選択・パラメータ・評価ロジック・他社APIの組み合わせ。  
  → 開発者向けの「パラメータ調整・データ解析」は **アキナイ側の関心事** として扱い、SFA の通常画面（IS・管理者向け）とは分けて考える。

---

## 2. 開発者UIの構想（パラメータ・データ解析など）

アキナイ（営業AI）を触る人（開発者・データサイエンティスト・運用チーム）向けの UI をどうするか。

### 2.1 想定する機能イメージ

| 領域 | 内容（例） |
|------|------------|
| **パラメータ調整** | 評価用モデル・温度・トークン上限、Whisper オプション、バッチ間隔など。環境別（dev/stg/prod）の切り替えも。 |
| **データ解析** | 架電結果・評価タグの集計、スコア分布、時系列。どのタグが効いているか・どの時間帯が良いか などの可視化。 |
| **実験・A/B** | プロンプトやパラメータのバリエーションを試し、結果を比較。 |
| **ログ・デバッグ** | AI 呼び出しログ、入力/出力のサンプル、エラー率。テナント・期間でフィルタ。 |
| **コスト・使用量** | トークン数・API 呼び出し回数・ストレージ。SaaS の課金とは別の「AI コスト」の可視化。 |

これらは **SFA の一般ユーザー（IS・管理者）には見せず、developer や特定ロールだけ** がアクセスする想定。

### 2.2 技術的な置き場所

- **SFA アプリ内の「/developer」のような専用ルート**  
  - 認証・テナントは SFA のまま。  
  - 画面は「パラメータ」「解析」「ログ」などタブ or サブページで分割。  
- **アキナイ側の別アプリ（スタンドアロン）**  
  - 営業AI 専用のダッシュボード。  
  - SFA の API を呼ぶか、アキナイ用の別 API を用意。  
  - 完全に分離できるが、認証・権限の共有は設計が必要。

「開発者UI」を SFA に含めるか、アキナイ側の別サービスにするかは、運用・権限・リリースサイクルで決める話。

## 3. Frame について（必要であれば検討）

- **ここでいう「Frame」**  
  - Cursor / 製品の「Frame」なのか、UI の「フレーム（レイアウト・iframe など）」なのか、文脈で解釈が分かれる。  
- **開発者UIで Frame が欲しくなる場面の例**  
  - 外部ツール（Notebook・Grafana・別ダッシュボード）の **埋め込み**。  
  - 複数パネル（パラメータ | グラフ | ログ）を **分割表示**。  
- **方針**  
  - まずは通常のダッシュボード（Next.js のページ＋コンポーネント）で足りるか検討。  
  - 埋め込み・複雑なレイアウトが必要になった時点で、**iframe や Frames（製品としての）の要否を検討**する、でよいと思う。

「必要であれば Frame を検討する」程度に留め、具体的な技術選定は要件が固まってからでよい。

## 4. 話し合いのポイント（整理用）

1. **SFA と アキナイ の境界**  
   - どこまでを SFA の「機能」として提供し、どこからを「アキナイ（営業AI レイヤー）」として別設計にするか。  
   - API の責務分離（SFA の API は「アキナイを呼ぶ窓口」だけ、中身はアキナイ側）。

2. **開発者UI のスコープ**  
   - パラメータ調整・データ解析・ログ・コストのうち、Phase でどこまでやるか。  
   - developer ロールだけで足りるか、それとも「AI 運用担当」のような別ロールが必要か。

3. **開発者UI のホスト**  
   - SFA 内の `/developer` で足りるか、最初からアキナイ側の別アプリにするか。  
   - 認証は SFA（NextAuth）の developer を流用するか、アキナイ用の別認証にするか。

4. **Frame / 埋め込み**  
   - 開発者UIで「外部ツールを埋め込みたい」「複数パネルを強く分けたい」という要件が出たタイミングで、Frame（製品）や iframe を検討する、でよいか。
