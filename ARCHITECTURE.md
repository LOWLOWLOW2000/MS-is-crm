# IS架電管理CRM アーキテクチャ概要

## 構成

- **apps/web**: Next.js 14 (App Router) — フロントエンド
- **apps/api**: NestJS — バックエンドAPI
- **apps/ai**: Python FastAPI — AI処理・バッチ（Phase2で本実装）

## AI評価機能（ZOOM録音 → 品質スコア）について

**現状（Phase1）**

- **AIスコアカード画面**（`/ai-score`）は **UIプレビューのみ** です。
- スコアの自動計算は行っておらず、表示はダミーまたは「未解析」です。
- ユーザーに期待感・将来像を伝える目的で、本番に近いレイアウトのみ先行実装しています。

**将来（Phase2）**

- AI評価は **バッチ処理** として実装予定です。
  - ZOOMクラウド録音を 1日2回（12:00・18:00）一括ダウンロード
  - Whisper で文字起こし → GPT-4o-mini で100タグ評価
  - 結果を `POST /ai/call-evaluations/batch` でAPIに登録
- フロントは `GET /reports/ai-scorecard` で取得した評価結果を表示します。
- 現時点では上記APIは **受け口のみ** で、保存処理は未実装です。

詳細は `.cursorrules` の「AI音声評価タグ設計」「ZOOM連携仕様」を参照してください。

## 構想: インサイドセールス向けAI解析・自動タグ付け（Zoom Phone + Gemini）

Zoom **Phone** の録音・文字起こし完了をWebhookで受け、Gemini 1.5 Flashで「個人情報マスク」「商談ランク」「100タグ判定」を一括実行してDB保存する実装タスクを構想に含める。

- **ドキュメント**: [docs/実装タスク-IS向けAI解析・自動タグ付け.md](docs/実装タスク-IS向けAI解析・自動タグ付け.md)
- 上記ドキュメントに、Webhookフロー・実装用プロンプト（System Instruction）・本プロジェクト（ZOOM Meeting / GPT-4o-mini バッチ）との対応を記載している。

## SaaS版IS-CRMにおけるデータ保持ポリシー（構想）

- **テナント側ストレージ**（Google Drive / 顧客S3 / 顧客Supabase 等）:
  - 通話音声・全文文字起こしなど、個人情報を含む可能性が高い「生データ」を保持する主体。
  - IS-CRMは、署名付きURLやアクセストークン経由で一時的に参照するのみを基本とする。
- **運営側Supabase**:
  - 会社情報（会社名・代表電話・住所などの法人情報）
  - ISメンバーの架電結果（結果タグ・次回架電日時など）
  - AIが付与した「100タグ評価」「スコア」「要約」などの**匿名化・抽象化された商談データ**
  - を最小限保持する。
- AI評価バッチ（`apps/ai`）は「テナント側ストレージの生データ → 匿名化＋タグ付け → 抽象化結果のみ `apps/api` 経由で保存」という流れを前提とし、将来的には運営UIからバッチ実行スケジュールや対象データ量を設定し、想定トークン数・推定コストを事前に確認できる設計を目指す。
